# PRD: Orion's Cookbook — Scout Camp Logistics App

Labels: enhancement, ready-for-agent
Status: ready-for-agent

---

## Problem Statement

Running food logistics for a scout camp is currently spread across notebooks, spreadsheets,
and group chats. The leaders (2–3 people) have to: rescale recipes by hand when the number of
campers changes, manually total up everything that needs buying, remember what's already in the
group's stores so they don't re-buy it, compare prices across shops, and keep a rough tally of
what was spent. The numbers drift, the math is error-prone, and nothing is connected — a change
to the menu doesn't update the shopping list, and what's in the cupboard isn't reflected in what
gets bought. It also has to work on a phone at a campsite where there is often no signal.

## Solution

**Orion's Cookbook** is a single self-hosted web app (installable PWA) that ties recipes, camp
menus, shopping, inventory, and spending together through one shared ingredient catalog.

- A **recipe cookbook** where changing the serving count rescales every ingredient by ratio.
- **Camp menus**: create a camp with dates and a headcount, then fill each day's five meal slots
  (Breakfast, Morning Snack, Lunch, Afternoon Snack, Dinner) with recipes that auto-scale to the camp.
- A **shopping list** generated from the camp's menu minus what's already in inventory, plus any
  manual items, with multiple shops and package prices per item so the app shows the cheapest shop,
  how many packages to buy, and the estimated cost.
- An **inventory** of the group's standing stores with par levels; low items can be pushed onto a
  camp's shopping list. Buying items lets you confirm the actual received quantity before it's added to stock.
- A per-camp **spending ledger** for recording actual expenses.

It runs in Docker on an always-on home server behind an existing Cloudflare tunnel, protected by a
single shared password. It works offline on phones at camp (single offline writer) and syncs back
on reconnect; when online, the 2–3 connected clients see each other's changes live.

## User Stories

### Recipes (cookbook)
1. As a camp cook, I want to create a recipe with a name, base serving count, and cooking steps, so that I can store how to make a dish.
2. As a camp cook, I want to add ingredients to a recipe by picking from a shared catalog with a quantity, so that the recipe links to real, reusable ingredients.
3. As a camp cook, I want to set how many servings a recipe makes by default, so that scaling has a baseline to scale from.
4. As a camp cook, I want to change the displayed serving count and have all ingredient quantities rescale by ratio, so that I don't recompute amounts by hand.
5. As a camp cook, I want quantities shown in sensible units (g/kg, ml/L, pieces), so that the numbers are readable at the scale I'm cooking.
6. As a camp cook, I want to categorize a recipe (breakfast/lunch/dinner/snack/dessert) and add free tags (vegetarian, campfire, quick), so that I can filter the cookbook.
7. As a camp cook, I want to edit and delete recipes, so that I can keep the cookbook current.
8. As a camp cook, I want to browse and search/filter recipes by category and tag, so that I can find a dish quickly.

### Ingredient catalog & inventory
9. As a quartermaster, I want a shared catalog of ingredients, each with a dimension (mass/volume/count) and base unit (g/ml/piece), so that recipes, inventory, and shopping all reference the same items.
10. As a quartermaster, I want to add non-food supplies (charcoal, bin bags) as count-based catalog items, so that I can track them alongside food.
11. As a quartermaster, I want each catalog item to record how much we currently have in stock, so that inventory reflects reality.
12. As a quartermaster, I want to manually adjust stock quantities, so that I can correct counts after a camp or a stocktake.
13. As a quartermaster, I want to set a par (minimum) level on an item, so that the app can flag when we're low.
14. As a quartermaster, I want low items (below par) flagged in the inventory view, so that I know what to restock.
15. As a quartermaster, I want to push low/needed inventory items onto a chosen camp's shopping list, so that restocking happens with the next shop.

### Shops & prices
16. As a shopper, I want to maintain a list of shops, so that I can record where we buy things.
17. As a shopper, I want to record, per shop, a package size and package price for an item, so that I can compare real-world offers (e.g. 1kg bag vs 2kg bag).
18. As a shopper, I want the app to compute price-per-unit from package size and price, so that I can compare shops fairly.
19. As a shopper, I want the app to flag the cheapest shop for each item, so that I buy at the best price.

### Camps & menus
20. As a camp planner, I want to create a camp with a name, start/end dates, and a headcount, so that I can plan its meals.
21. As a camp planner, I want each day of the camp laid out with five ordered meal slots (Breakfast, Morning Snack, Lunch, Afternoon Snack, Dinner), so that I plan meals in a familiar structure.
22. As a camp planner, I want to add one or more recipes to any meal slot, so that a meal can have a main plus a dessert.
23. As a camp planner, I want recipes in the menu to auto-scale to the camp headcount, so that quantities match the number of eaters.
24. As a camp planner, I want to override the serving count on an individual recipe instance, so that a hike-day meal can feed fewer people.
25. As a camp planner, I want to edit and remove recipes from slots, so that I can adjust the plan.
26. As a camp planner, I want to see the full week's menu at a glance, so that I can sanity-check variety and balance.

### Shopping list
27. As a shopper, I want a shopping list generated from a camp's menu by summing all scaled ingredients, so that I don't total amounts by hand.
28. As a shopper, I want what we already have in inventory subtracted from the needed amounts, so that I only buy the shortfall.
29. As a shopper, I want to add manual items to the shopping list (things not in any recipe), so that I capture supplies like charcoal.
30. As a shopper, I want to regenerate the shopping list after changing the menu or inventory, so that it stays accurate.
31. As a shopper, I want each shopping line to suggest the cheapest shop, the number of packages to buy, and the cost, so that I can shop efficiently.
32. As a shopper, I want an estimated total cost for the shopping list, so that I have a budget expectation.
33. As a shopper, I want to mark items as bought, so that I can track shopping progress.
34. As a shopper, I want to confirm the actual quantity received (editable, may differ from planned) when marking an item bought, so that inventory reflects what really arrived.
35. As a shopper, I want confirmed purchases to add the received quantity to inventory stock, so that our stores stay current without re-entry.

### Spending
36. As a treasurer, I want to log expenses against a camp (amount, label, optional category, optional day), so that I have a record of what was spent.
37. As a treasurer, I want to see the total spent for a camp, so that I can report costs.
38. As a treasurer, I want to edit and delete expense entries, so that I can fix mistakes.

### Offline, realtime & access
39. As a leader at a campsite with no signal, I want the app to keep working offline on my phone, so that I can view recipes/menus and record changes.
40. As a leader, I want changes I make offline to sync to the server when I'm back online, so that nothing is lost.
41. As a leader, I want the other 1–2 leaders' changes to appear live on my screen when we're all online, so that we stay in sync without refreshing.
42. As a leader, I want to install the app to my phone's home screen, so that it feels like a native app.
43. As a leader, I want the app protected by a single shared password, so that strangers who find the public URL can't read or edit our data.
44. As a leader, I want my device to stay logged in (including offline), so that I don't re-enter the password constantly.
45. As an operator, I want the whole app to run as a single Docker container with a persistent database volume behind our Cloudflare tunnel, so that hosting is simple.
46. As an operator, I want prices and units shown in RON and metric, so that they match our context.

## Implementation Decisions

### Architecture
- **Monorepo (pnpm workspace)** with three packages: `packages/core` (shared pure domain logic + Zod types/schemas), `apps/server` (Fastify + Prisma + `ws`), `apps/web` (React + Vite PWA).
- **Single Docker image**: Fastify serves the built web assets, the JSON REST API, and the WebSocket endpoint. SQLite database file on a mounted volume. cloudflared runs separately (already configured by the operator).
- **TypeScript end-to-end.** Currency RON, metric units, single currency app-wide.

### Deep modules (pushed into `packages/core`, pure and isolation-testable)
- **units** — dimension/base-unit conversion and display. Interface: `toBase`, `fromBase`, `format(qty, dimension)`. Handles g↔kg and ml↔L display thresholds; count stays as pieces.
- **scaling** — `scaleRecipe(recipe, targetServings)` returns ingredient lines at `baseQty × targetServings / baseServings`, with display rounding.
- **needs** (the heart) — `computeToBuy(menuEntries, recipes, headcount, inventory)`: for each ingredient, sum `recipeIngredient.qty × (servingsOverride ?? headcount) / recipe.baseServings` across all menu entries, then `toBuy = max(0, needs − stockQty)`.
- **pricing** — `priceLine(toBuy, shopPrices)` → cheapest shop by `packagePrice / packageSize`, `packages = ceil(toBuy / packageSize)`, `cost = packages × packagePrice`; `estimateTotal(...)` sums lines.

### Other modules
- **sync** (`apps/web`) — Dexie (IndexedDB) mirror of all server data + an outbox of pending write ops. Interface: `enqueue(op)`, `flush()`, `subscribe()`. Reads come from Dexie (offline-capable); writes apply optimistically to Dexie and enqueue; if online, send immediately, else queue. On reconnect, replay outbox. Designed against an injectable transport so it can be tested without a real network.
- **realtime** (`apps/server`) — WebSocket hub: `broadcast(change)` to all connected clients, fan-out on each successful mutation.
- **auth** (`apps/server`) — shared-secret token: `issue()` after a correct password check, `verify(token)` (HMAC of `APP_SECRET`). All REST/WS routes require a valid token. Password supplied via env (`APP_PASSWORD`).
- **repository** (`apps/server`) — thin Prisma-backed CRUD per entity; the I/O shell around the pure core.

### Sync & conflict model
- **Single offline writer** (the camp device) is an assumed invariant → **last-write-wins** using a client-supplied timestamp on each mutation. No CRDTs or operation-merge logic.
- Service worker (Workbox / vite-plugin-pwa) caches the app shell so the PWA boots offline.

### Schema (Prisma / SQLite)
- **Ingredient**: `id, name, dimension (MASS|VOLUME|COUNT), baseUnit (g|ml|piece), category, stockQty, parLevel?, notes?` — serves as catalog, inventory, and supplies.
- **Shop**: `id, name`.
- **ShopPrice**: `id, ingredientId, shopId, packageSize (base units), packagePrice (RON)`.
- **Recipe**: `id, name, baseServings, category, tags (string[]), steps (ordered string[])`.
- **RecipeIngredient**: `id, recipeId, ingredientId, quantity (base units)`.
- **Camp**: `id, name, startDate, endDate, headcount, notes?`.
- **MenuEntry**: `id, campId, date, slot (BREAKFAST|MORNING_SNACK|LUNCH|AFTERNOON_SNACK|DINNER), recipeId, servingsOverride?`.
- **ShoppingItem**: `id, campId, ingredientId, source (AUTO|MANUAL|RESTOCK), neededQty (base units), chosenShopId?, purchased (bool), purchasedQty? (base units)`.
- **Expense**: `id, campId, date?, amount (RON), label, category?`.

### Behavior clarifications (decided during interview)
- Menus reference recipes **live** (no per-camp recipe snapshot); regenerating the shopping list recomputes from current recipes/inventory.
- Cooking does **not** auto-decrement inventory; post-camp stock is corrected manually.
- Spending is a standalone manual ledger; the shopping list's estimated cost is shown separately and is not auto-posted as an expense.
- Any recipe may be placed in any meal slot; category is used for filtering/suggestion, not as a hard constraint.

## Testing Decisions

A good test here verifies **external behavior through a module's public interface**, not its internals.
The pure `core` modules take plain inputs and return plain outputs, so tests assert results for given
inputs with no mocks. The `sync` engine is tested against a fake/in-memory transport, asserting
observable outcomes (what ends up in the mirror, what the server receives on flush), not private state.

Modules to be tested (confirmed with the user):
- **needs** — aggregation across multiple recipes/slots, serving overrides, inventory subtraction, the `max(0, …)` floor, mixed dimensions.
- **scaling** — ratio correctness, scaling up and down, rounding/display behavior, base-serving edge cases.
- **pricing** — cheapest-shop selection by price-per-unit, `ceil` package math, total estimation, items with no price.
- **units** — to/from base conversion and g/kg, ml/L display thresholds; count handling.
- **sync** — outbox enqueue/flush against a fake transport, offline-then-reconnect replay, last-write-wins on a timestamp clash.

Prior art: none yet (greenfield). Establish the convention with these as the reference: Vitest,
pure function tests in `packages/core`, and an injectable-transport test for `sync`. Later modules
should follow this external-behavior style.

## Out of Scope

- Per-user accounts, roles, or permissions (single shared password only).
- Multi-writer offline conflict resolution / CRDTs (single offline writer is assumed).
- Recipe photos, prep/cook times, and free-form recipe notes.
- Cooking units (cups, tbsp, tsp) and density-based conversions.
- Auto-decrementing inventory when meals are cooked.
- Budgets, auto-posting purchases as expenses, expense splitting / who-paid tracking.
- Per-camp recipe snapshots / historical recipe versioning.
- Dietary subgroup counts (e.g. scaling a meal to a vegetarian sub-count).
- Multi-currency and imperial units.
- Cloudflare Access / edge auth (the in-app shared password is the chosen mechanism).
- Native mobile apps (PWA only).
